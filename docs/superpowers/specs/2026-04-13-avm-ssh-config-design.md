# `avm ssh-config` Design

## Goal

Let users type `ssh avm-<id>` directly and reach the container — no port flags,
no manual config edits per container. The immediate motivation is Warp's
"warpify" detection, which keys off the literal `ssh <host>` invocation, but
the feature is generally useful for any `ssh`-based tooling (scp, rsync,
editor remote modes, etc.).

## Principles

- **`~/.ssh/config` is user-owned.** avm touches it only when the user asks,
  and only within clearly delimited markers.
- **`~/.avm/ssh_config` is avm-managed.** Regenerated wholesale; users don't
  edit it.
- **No daemons, no watchers.** The managed file is refreshed at known events
  (`create`, `clean`) and on demand (`ssh-config sync`).
- **Preferences persist.** If a user declines the install prompt, we don't
  pester them again.

## Commands

### `avm ssh-config [sync]`

Regenerate `~/.avm/ssh_config` from the current set of avm containers.
Idempotent. Default action when no subcommand is given.

**Flow:**
1. `listAvmVms()` to enumerate containers.
2. For each container with an `avm.ssh-port` label, emit a `Host avm-<id>`
   block (see file format below). Containers without a port label (pre-SSH
   era) are skipped.
3. Write the file atomically (write to `~/.avm/ssh_config.tmp`, rename).

**Errors:**
- If `~/.avm/` doesn't exist, create it.
- If the managed file can't be written, print the error and exit non-zero.

### `avm ssh-config install`

Ensure `~/.ssh/config` includes `~/.avm/ssh_config`, and that the managed
file exists.

**Flow:**
1. Call `ssh-config sync` to ensure the managed file exists.
2. Read `~/.ssh/config` (create if missing, with `0600` perms).
3. If the avm marker block is already present → no-op.
4. Else if a bare `Include ~/.avm/ssh_config` line exists (user added it
   manually) → print a note and treat as installed (no file mutation;
   leave the user's line untouched).
5. Else prepend the marker block (see format below).
6. Write state: `sshConfig.installPrompt = "installed"`.

**Idempotent.** Safe to run repeatedly.

### `avm ssh-config uninstall`

Remove avm's marker block from `~/.ssh/config`. Leaves `~/.avm/ssh_config`
in place (harmless without the Include).

**Flow:**
1. Read `~/.ssh/config`.
2. If the marker block exists, remove it (including the lines between
   markers, plus a single trailing blank line if present).
3. If no marker block exists → no-op, print a message.
4. Clear `sshConfig.installPrompt` from state so a fresh `install` can
   prompt again.

**Never** touches an `Include` line the user added outside our markers.

## File Formats

### `~/.avm/ssh_config`

```
# Managed by avm. Do not edit — regenerated by `avm ssh-config sync`.

Host avm-<id1>
  HostName localhost
  Port <port>
  User agent
  StrictHostKeyChecking no
  UserKnownHostsFile /dev/null

Host avm-<id2>
  ...
```

One block per container. Matches the output of `avm ssh --print-config`
so the behavior is consistent.

### `~/.ssh/config` marker block

```
# >>> avm managed >>>
Include ~/.avm/ssh_config
# <<< avm managed <<<
```

Prepended to the file (before any existing content) so avm hosts resolve
before any global `Host *` defaults the user may have set.

## State File

New convention: `~/.avm/state.json`.

**Schema (current):**
```json
{
  "sshConfig": {
    "installPrompt": "installed" | "declined"
  }
}
```

`installPrompt` absent means the user has never been asked.

**Access pattern:** `lib/state.ts` exposes `readState()` and
`updateState(partial)`. Updates are read-modify-write with a shallow merge
per top-level key, so future additions don't clobber unrelated keys.
JSON is written with 2-space indent, trailing newline.

## Prompt Integration

### `avm create`

After post-creation setup completes and before the final success message:

1. Always call `syncSshConfig()` (cheap; keeps file current whether or
   not the user has installed the include).
2. If `state.sshConfig.installPrompt` is absent:
   - Prompt (via `@clack/prompts` `select`):
     ```
     Enable `ssh avm-<id>` shortcut by adding an Include to ~/.ssh/config?
       ❯ Yes, install it
         Not now (ask again next time)
         No, don't ask again
     ```
   - "Yes" → run `ssh-config install`, set `installPrompt: "installed"`.
   - "Not now" → no state change.
   - "No" → set `installPrompt: "declined"`.
3. If `installPrompt` is `"installed"` or `"declined"`, skip the prompt.

### `avm clean`

After removing containers, call `syncSshConfig()` to drop the stale
`Host` blocks. No prompt logic here.

## Implementation

### New files

- `cli/commands/ssh-config.ts` — citty subcommand group (`sync`, `install`,
  `uninstall`), registered in `cli/avm.ts`.
- `lib/ssh-config.ts` — pure functions used by the command and by
  `create`/`clean`:
  - `renderManagedFile(vms): string`
  - `syncSshConfig(): Promise<void>` (writes the managed file)
  - `installInclude(): Promise<{ changed: boolean }>`
  - `uninstallInclude(): Promise<{ changed: boolean }>`
- `lib/state.ts` — `readState()`, `updateState(partial)`, typed schema.

### Modified files

- `cli/avm.ts` — register `ssh-config` subcommand.
- `cli/commands/create.ts` — call `syncSshConfig()` and run the prompt.
- `cli/commands/clean.ts` — call `syncSshConfig()` after removal.
- `lib/config.ts` — export paths for `avmSshConfigFile` and `avmStateFile`.
- `README.md` — document the new command and mention first-time setup.
- `skills/avm/SKILL.md` — describe the `ssh avm-<id>` shortcut.
- `templates/vm-claude.md` — no changes (in-container behavior unaffected).

## Out of Scope

- Auto-installing the Include without user consent.
- Watching container state changes (no daemon).
- Syncing on `start`/`stop` (ports are stable across restarts).
- Managing SSH known_hosts — we continue to set
  `UserKnownHostsFile /dev/null` since container host keys are ephemeral.
- Replacing `avm ssh` — that command continues to work standalone for
  users who don't install the include.

## Risks and Mitigations

- **User's `~/.ssh/config` is sensitive.** Mitigation: mutations only
  inside the marker block; `uninstall` removes only what `install` added;
  user-added Include lines (outside our markers) are never modified.
- **OpenSSH version skew.** `Include` requires OpenSSH 7.3 (2016-08).
  Modern macOS and Linux ship newer versions. No runtime check — if
  users hit this, an `ssh -V` mention in the README is sufficient.
- **Stale entries for removed containers.** Mitigation: `clean` triggers
  `sync`. If a user removes a container via `docker rm` directly, the
  file will be stale until the next `sync`. Acceptable — the stale
  entry just fails to connect.
- **Pre-SSH-era containers.** Skipped cleanly in the renderer; no error.
